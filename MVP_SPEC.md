# Codex Level — Build Week MVP Spec

## 1. 产品目标

Codex Level 是一个 macOS 菜单栏应用，把用户在 Codex 中已经形成的长期使用记录，转化为持续可见的成长身份。

核心叙事：

> CodexBar 回答“现在还能用多少”，Codex Level 回答“我和 Codex 一起积累了多少”。

MVP 只验证一个问题：用户看到自己的真实累计 Token、等级和升级进度后，是否会产生持续使用和身份认同。

## 2. 目标用户与前提

- 使用 macOS；
- 已在本机登录 Codex；
- 本机存在可用的 Codex ChatGPT 登录态；
- 接受 Codex Level 在明确授权后只读访问本机 Codex 认证文件。

未登录、API Key 模式、接口未开放或认证失效时，应用必须说明原因，不伪造数据。

## 3. MVP 用户体验

菜单栏默认显示：

`Codex Lv.11 · 32%`

点击后只展示：

1. 周额度已使用比例；
2. 下次重置时间；
3. 账号累计 Token；
4. 当前等级；
5. 距离下一级的进度；
6. 数据不可用时的明确错误与重试入口。

MVP 不展示 5 小时额度。

## 4. 数据来源与口径

### 账号累计 Token

- 请求：`GET https://chatgpt.com/backend-api/wham/profiles/me`
- 字段：`stats.lifetime_tokens`
- 本机已验证该字段与 Codex Desktop 个人资料中的“累计 Token 数”一致。

该接口同时可能返回峰值 Token、连续天数、最长任务、任务数和活动分布，但除累计 Token 外，均不进入 MVP 核心界面。

### 周额度

- 首选：本机 `codex app-server` 的 `account/rateLimits/read`；
- 使用主窗口的 `usedPercent`、`windowDurationMins` 和 `resetsAt`；
- 当前窗口不存在时展示“暂不可用”，不推算或伪造。

### 认证

- 只读使用 `~/.codex/auth.json` 中的登录态；
- 不记录、不展示、不上传 Access Token 或 Refresh Token；
- 不把认证信息写入日志、分析系统或崩溃报告；
- 认证过期时提示用户重新登录 Codex。

产品文案使用“账号累计 Token”，不使用“能力值”“生产力”或“节省时间”等未经证明的含义。

## 5. 等级约束

- 等级由累计 Token 经过确定、单调递增的规则计算；
- 相同累计 Token 必须始终得到相同等级；
- 第一版只展示数字等级和升级进度；
- 星星、月亮、太阳、皇冠属于视觉层，不改变底层等级；
- 等级阈值表必须由产品负责人确认后才能实现；
- 不在 MVP 中设计动态难度、赛季、排行榜或防刷机制。

## 6. 明确不做

- 勋章或通用成就系统；
- 分享卡、排行榜和社交关系；
- 多 Provider 支持；
- 自建账号、后端、数据库或云同步；
- 全量 Session 日志扫描；
- 成本估算和生产力评估；
- Widget、CLI、浏览器扩展；
- 通知系统和自动更新；
- 为未来需求预先设计插件系统或通用框架；
- 复制 CodexBar 的实现或多 Provider 架构。

## 7. 失败与降级

- 找不到 Codex 登录态：提示“请先登录 Codex”；
- Profile 接口失败：累计 Token 显示不可用，并允许重试；
- app-server 启动或额度读取失败：等级仍可显示，额度区域单独降级；
- 内部接口字段缺失：不崩溃、不显示 `0` 冒充真实值；
- 保留最近一次成功数据时，必须标记更新时间和“可能已过期”。

## 8. MVP 验收标准

- 已登录测试账号能够读取与 Codex Desktop 个人资料一致的累计 Token；
- 能够读取并展示周额度和重置时间；
- 菜单栏能够稳定显示等级与升级进度；
- Token、账号标识和响应正文不会出现在应用日志中；
- 断网、未登录、Token 失效、字段缺失和 app-server 失败均不会导致崩溃；
- 应用重启后可以重新获取数据；
- 整个项目不包含数据库、后端、多 Provider 抽象或成就引擎。

## 9. 开发方式与复杂度红线

### 参考优先规则

每个开发切片开始前，先确认是否已有可靠实现可供学习：

1. 先检查本机 Codex 能力和官方文档；
2. 再检查 CodexBar 等成熟开源项目中与当前切片直接相关的实现；
3. 记录采用了什么思路、拒绝了什么复杂设计；
4. 只实现当前验收标准所需的最小部分，不照搬完整架构；
5. 未经确认，不新增第三方依赖，也不为单一实现创建通用抽象。

“参考优先”表示不凭空假设接口、协议和平台行为，不表示机械复制开源代码或为每一行代码进行搜索。

按垂直切片开发，每阶段完成后停止并验收：

1. **数据切片**：只验证登录态、累计 Token、周额度；
2. **核心切片**：加入菜单栏、弹出面板、数字等级和进度；
3. **可靠性切片**：补齐失败状态、隐私保护和重试；
4. **发布切片**：签名、公证和发布包装，单独决策。

实现期间未经产品负责人确认，不得新增功能、依赖、数据源、持久化方案、通用抽象或发布目标。

## 10. MVP 成功判定

满足以下两点即可认为 Build Week MVP 成立：

1. 技术上，目标用户安装后能在授权范围内自动读取自己的真实累计 Token；
2. 产品上，测试用户能够理解等级代表“与 Codex 的累计使用历程”，并愿意再次打开查看进度。

勋章系统是否进入下一阶段，只依据 MVP 用户反馈决定，不作为首版完成条件。
